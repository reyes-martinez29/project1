Regla de oro

Haz cambios pequeños y commitea seguido. Un commit debe representar una idea clara: “agregué X” o “corregí Y”.

Flujo típico

  1. 1 Editar archivos
  2. 2 git status
  3. 3 git add .
  4. 4 git commit -m "..."
  5. 5 git push

Tip: si algo sale mal, revisa primero git status.

1) Introducción: Git vs GitHub

Git

Git es un sistema de control de versiones. Te permite guardar “fotografías” (commits) de tu proyecto, comparar cambios, volver atrás y trabajar con ramas sin romper lo que ya funciona.

  • Funciona localmente en tu computadora
  • Te da historial, ramas, merges y herramientas para colaborar

GitHub

GitHub es una plataforma para alojar repositorios Git en la nube y colaborar. Lo importante aquí: repos remotos, Pull Requests, Issues, revisiones y automatización (CI).

  • Tu repo remoto (online)
  • PRs para integrar cambios con revisión
  • Issues para organizar tareas
Idea clave: Git es la herramienta. GitHub es el lugar donde compartes y colaboras.

2) Instalación y configuración inicial

Antes de usar Git en serio, configura tu identidad (nombre y correo). Eso se guarda dentro de cada commit. Después, verifica que Git esté instalado y aprende 2–3 atajos útiles.

Configurar nombre y correo

Esto firma tus commits. Hazlo una vez.

git config --global user.name "Tu Nombre"
        git config --global user.email "tu_correo@ejemplo.com"

Verificar tu configuración

git config --list

Config útil (opcional)

# Editor para mensajes de commit (VS Code)
        git config --global core.editor "code --wait"

        # Log bonito (atajo)
        git config --global alias.lg "log --oneline --decorate --graph --all"

        # Comprobar tu alias
        git lg

Un alias es solo un atajo para un comando largo. Si algo no te funciona, ejecuta el comando completo.

Comprobar instalación

git --version

Ayuda rápida

git --help git 
        help commit
Tip: si estás en WSL, Git normalmente ya viene instalado o se instala con el gestor de paquetes.

3) Flujo básico: init → add → commit

Crear un repositorio

Esto inicializa Git en la carpeta actual y crea la carpeta .git (ahí vive todo el historial). 

git init

Ver estado

Te dice qué archivos cambiaron, cuáles están listos para commit y cuáles no. 

git status

Stage: preparar cambios

El “stage” es una bandeja donde eliges qué entra al siguiente commit. Útil si hiciste varios cambios pero quieres commitearlos por separado. 

# Agregar todo
          git add .

          # Agregar un archivo específico
          git add index.html

          # Agregar varios archivos
          git add index.html styles.css

Crear un commit

Un commit debe representar una idea clara. Evita mensajes vagos como “update”. Mejor: qué hiciste y por qué. 

git commit -m "feat: add workflow section"

Ejemplos de buenos mensajes

  • feat: add intro section
  • style: improve hero responsive layout
  • fix: correct anchor ids in navbar
  • docs: update README with usage
Receta rápida: 
# 1) ver qué cambió
        git status

        # 2) revisar diferencias (opcional pero recomendado)
        git diff

        # 3) preparar cambios
        git add .

        # 4) crear commit
        git commit -m "feat: describe your change"
